# PROJECT_RULES.md — 開発ルール

> このファイルはプロジェクトの全開発者（AIを含む）が守るべきルールを定義します。
> コードを変更する前に必ず確認してください。

---

## 1. 大前提：既存機能を壊さない

- 変更前に影響範囲を必ず確認する
- 関数を削除・リネームする場合は全参照箇所を検索してから実行する
- `render()` の呼び出しパターン（`setupDelegation` + `bindDynamic`）を崩さない
- スクリプトロード順序（`db.js → constants.js → utils-core.js → state.js → delegation.js → app.js`）を変更しない
- 変更後はブラウザでダッシュボード・商品一覧・商品詳細・ラベル編集の動作を確認する

---

## 2. アーキテクチャルール

### グローバル状態
- 状態変数は `src/state.js` のみで宣言する
- 新しい状態が必要な場合は `state.js` の末尾に追加する
- `localStorage` への読み書きは `safeGet() / safeSet() / safeDel()` を使用し、直接の `localStorage.*` 呼び出しは禁止
- OpenAI API キーは **sessionStorage のみ** に保存する（`localStorage` 厳禁）

### イベント処理
- クリック・キーダウン・input・change イベントは `src/delegation.js` の `setupDelegation()` に集約する
- `render()` のたびに `addEventListener` を追加してはいけない（メモリリーク）
- `render()` ごとに再バインドが必要な処理は `bindDynamic()` に書く

### レンダリング
- UI 更新は原則 `render()` で行う（DOM を直接操作する場合は `showStatus()` / `updateMasterSaveStatus()` のような独立関数のみ）
- デバウンスが必要な更新は `scheduleRender()` を使う
- バージョン文字列（`?v=20260709m`）は CSS / JS を変更したら必ず更新する（全ファイル一括）

---

## 3. コード品質

### 関数設計
- 1関数は1つの責務のみ（Single Responsibility）
- 100行を超える関数は分割を検討する
- HTML生成関数は `〇〇Html()` という命名にする
- 巨大な template literal は変数に分割して可読性を高める

### コメント
- コメントは「なぜ（WHY）」だけ書く。「何を（WHAT）」はコードが語る
- 一時的な回避策やバグ対応は必ずコメントで理由を記録する
- TODO コメントは `// TODO: 説明` 形式で書き、`TODO.md` にも登録する

### 共通化
- 同じロジックが3箇所以上あれば関数化する
- HTML のボイラープレートは `saasLayout()` や `emptyProduct()` のようなヘルパーを作る
- 定数は `src/constants.js` に集約する（マジックナンバー禁止）

### 命名
- 変数・関数はキャメルケース（`masterFilter`, `calcCompletion`）
- HTML クラス名はケバブケース（`master-card`, `filter-active-tag`）
- boolean は `is〇〇` / `has〇〇` / `can〇〇` で始める
- HTML 生成関数は `〇〇Html()` suffix

## 4. UX ルール

### 食品メーカーが毎日使う前提で設計する

- **説明書なしで使える** — 初めて使う食品担当者が迷わないこと
- **クリック数を最小化** — 目標の操作を 3 クリック以内で完了できること
- **フィードバックを即座に返す** — 保存・削除・AI 処理はすべてトーストやステータスで結果を通知する
- **破壊的操作は undo できる** — 削除・全置き換えなどは 5 秒以内に取り消せるようにする
- **モーダルの背景クリックは安全** — 背景クリックは「キャンセル」であり、破壊的アクションを実行してはいけない

### 空状態の設計
- データが 0 件の場合は次のアクションを案内する（ただの「ありません」は禁止）
- フィルター結果 0 件は「フィルターをクリア」ボタンを必ず表示する

### AI 機能
- API キー未設定でも使える（テンプレート回答にフォールバック）
- AI 処理中はローディング表示をする
- AI 処理失敗時は具体的なエラーメッセージを表示する
- AI で自動化できる繰り返し作業は積極的に実装する

---

## 5. UI ルール

### デザインシステム
- カラーはデザイントークン（`--blue`, `--text`, `--border` 等）を使用する
- 角丸は `--radius-sm: 8px` / `--radius: 12px` / `--radius-lg: 16px` から選ぶ
- シャドウは `--shadow-sm` / `--shadow` / `--shadow-md` / `--shadow-lg` を使う
- フォントサイズ: 本文 14px / 小テキスト 12px / ラベル 11px

### シンプルさの優先
- 1 画面に情報を詰め込まない（スクロールより画面分割）
- ボタンは目的が明確な動詞で書く（「する」で終わる）
- アイコンと文字を必ず組み合わせる（アイコンのみは禁止）

### レスポンシブ
- ブレークポイント: 560px（スマホ）/ 768px（タブレット）/ 900px（デスクトップ）
- スマホでは重要でない要素を非表示にして良い（`.kbd-hints` 等）
- タッチターゲットは最小 44px 以上

---

## 6. アクセシビリティ

- インタラクティブな `div/span` には `role="button"` + `tabindex="0"` を付ける
- `role="button"` 要素は Enter/Space キーで操作できること（`delegation.js` で委譲済み）
- 画像には `alt` 属性を必ず付ける
- フォームのラベルは `<label>` と `<input>` を明示的に紐付ける
- フォーカスインジケーターを `outline: none` で消すことは禁止（代替の `focus-visible` スタイルを必ず設定する）
- カラーのみで状態を伝えない（アイコン・テキストを必ず併用する）

---

## 7. セキュリティ

- ユーザー入力は必ず `escapeHtml()` でエスケープしてから HTML に埋め込む
- `innerHTML` に直接ユーザー入力を埋め込むことは厳禁
- OpenAI API キーは `sessionStorage` のみに保存する（ページを閉じると消える）
- API キーをコンソールログや HTML に出力しない
- ファイルインポート時はパースエラーを必ず try-catch で処理する
- 外部 URL をユーザー入力から構築する場合はバリデーションを行う

---

## 8. パフォーマンス

- `derive(p)` は 1 レンダリングサイクルで同一商品について 1 回のみ呼ぶ
- `calcTodo()` には `derivedAll` を渡して再計算を防ぐ
- `render()` は全 DOM を再構築するため、高頻度で呼ばないこと（デバウンスを使う）
- 画像は `imageDataUrl`（base64）で保存するが、サイズ超過の場合はリサイズして保存する

---

## 9. バージョン管理フロー

修正を行った場合、以下の順番でファイルを更新する:

1. **コードを修正** する
2. `index.html` の **バージョン文字列を更新**（`?v=YYYYMMDD[アルファベット]`）
3. **ブラウザで動作確認**（構文エラー・コンソールエラーを確認）
4. `CHANGELOG.md` に **変更内容を記録** する
5. `TODO.md` の **完了タスクを ✓ に更新** し、新しい改善点を追加する
6. 必要に応じて `ROADMAP.md` を更新する

---

## 10. 禁止事項

| 禁止 | 理由 |
|------|------|
| `localStorage` への API キー保存 | セキュリティリスク |
| `innerHTML = ユーザー入力` | XSS リスク |
| `showModal` の背景クリックで破壊的処理 | 意図しないデータ消失 |
| `addEventListener` を `render()` 内で呼ぶ | メモリリーク |
| `outline: none` でフォーカスを消す | アクセシビリティ違反 |
| マジックナンバーをコード内に直書き | 保守性低下 |
| 関数内で `derive(p)` を複数回呼ぶ | パフォーマンス問題 |
| `state.js` 以外で状態変数を宣言 | 状態管理の散乱 |
